Abstract

The RMap ontology supports the description of a distributed scholarly work. Maps of works are described in the form of an rmap:DiSCO - a simple, lightweight version of an OAI-ORE Aggregation - with provenance information captured in an RMap Event, and DiSCO creators described as RMap Agents.

Introduction

Today's scholarly works can be dynamic, distributed, and complex. They can consist of multiple related components (article, dataset, software, multimedia, webpage, etc.) that are made available asynchronously, assigned a range of identifiers, and stored in different repositories with uneven preservation policies. The RMap ontology supports the description of the components of distributed works and, optionally, the context around it. The core of the RMap model is the RMap DiSCO (Distributed Scholarly Compound Object), which draws on concepts from OAI-ORE to provide a lightweight Aggregation object that describes distributed scholarly works. To capture provenance of DiSCOs, there is a simple Event model. Agents generate Events and can act upon both Agents and DiSCOs. The RMap ontology drives the RMap system (https://rmap-hub.org).

Description

The diagram below shows one way in which RMap classes relate to each other. The example shows a DiSCO creation. There are several other rmap:eventTypes that define the type of relationship between the rmap:Event and the rmap:Object that it acts upon. Additional statements can be added about the aggregated rdf:Resources using any ontology as appropriate, provided the statements form a connected graph with those rdf:Resources. Agents can also be generated by an Event, and an Agent can create their own Agent record: For RDF examples of Events, DiSCOs and Agents, visit the RMap wiki: https://rmap-project.atlassian.net/wiki/spaces/RMAPPS/pages/5472305/Media+Types

aggregatesop back to ToC or Object Property ToC

IRI: http://www.openarchives.org/ore/terms/aggregates

One or more rdfs:Resource can be aggregated by an rmap:DiSCO. The aggregated resources represent components of a scholarly object that are related to each other to form a compound representation of a work. A component can, for example, be a person, organization, dataset, webpage, article, sample, code, or anything else related to a scholarly work. Ideally, additional statements will be added to describe the components and their relationship to one another, but this is not a requirement.

derived objectop back to ToC or Object Property ToC

IRI: http://purl.org/ontology/rmap#derivedObject

The rmap:DiSCO that was derived as a result of an update or derivation event against a DiSCO. DiSCOs are immutable, so when a DiSCO is updated, an Event indicates the location of the previous version as well as the new DiSCO that was derived from it (the derivedObject). A derivedObject can be generated under two Event types: (1) an update DiSCO Event occurs when the update is performed by the same rmap:Agent that created the original DiSCO, the previous version is inactivated (include inactivatedObject in the Event) and the new version is the derivedObject. (2) A derivation Event type occurs when an Agent that did not originally create a DiSCO generates a new version of it. In this instance the previous version stays the same as it was and the URI of that should be indicated using hasSourceObject. The new DiSCO is the derivedObject.

inactivated objectop back to ToC or Object Property ToC

IRI: http://purl.org/ontology/rmap#inactivatedObject

The rmap:DiSCO inactivated as a result of an rmap:Event. Two kinds of Events result in an inactivatedObject property: (1) an inactivation Event marks a DiSCO as inactive, meaning the data can still be accessed through public interfaces, but it may not be accurate or up to date. (2) an update Event is initiated against a DiSCO by the original Agent that created the DiSCO. The target version becomes inactive (the inactivatedObject) and the new version is the derivedObject.

has range

lineage progenitorop back to ToC or Object Property ToC

IRI: http://purl.org/ontology/rmap#lineageProgenitor

The rmap:lineageProgenitor property references the first rmap:DiSCO or rmap:Agent URI in a lineage. If, for example, a DiSCO is created, updated, updated again, then tombstoned, 3 different DiSCO URIs will exist in the object's lineage (one for each update) and 4 rmap:Events. The rmap:lineageProgenitor of the Event will always contain the URI of the original DiSCO created. This can be used to support the retrieval of related Events and versions.

user auth idop back to ToC or Object Property ToC

IRI: http://purl.org/ontology/rmap#userAuthId

A calculated identifier that relates to the user's authorization path. This can be used to support individual verification that a person is the Agent they claim to be without publicly displaying e.g. email information. For example, in the RMap system, this contains a URI constructed from the SHA256 encoding of the identityProvider+accountId e.g. http://orcid.org + {orcidid}. Since minimal information is captured in an rmap:Agent, and oauth users can enter any name they like, this provides a method to individually verify an Agent's identity by confirming this field matches their account information.
